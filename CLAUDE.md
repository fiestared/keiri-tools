# keiri-tools — 経理ミニツールズ（柱A: マイクロWebツール）

AI月30万円プロジェクト・柱Aの初号プロダクト群。経理実務の「ちょっとした計算」を
静的サイト(サーバー処理なし・データ送信なし)で提供し、広告+将来のPro機能で収益化する。
選定根拠: ニッチ調査(gbrain research/jp-micro-webtool-niche-2026-07)で
「振込手数料 先方負担計算機の不在」が唯一敵対的検証を生き残った空白だったため。

## 構成
- `docs/` — 静的サイト一式(そのままデプロイ可能)
  - `index.html` — ハブトップ(ツール一覧)
  - `senpou-futan/` — 振込手数料「先方負担」計算 + 入金差額チェック
  - `assets/senpou_core.js` — 計算純ロジック(DOM非依存)。**変更したら必ずテストを回す**
  - `ext/amazon-receipt/selectors.json` — Chrome拡張のリモートセレクタ定義(下記)。
    AmazonのDOM変更はこのJSONの更新+pushだけで直せる(ストア再審査不要)
- `extension/amazon-receipt/` — アマゾン領収書・電帳法索引簿メーカー(Chrome拡張, MV3)。
  設計・現状・動作確認手順は同フォルダのREADME.md。セレクタは実DOM未検証(フィクスチャ待ち)
- `tests/test_senpou.mjs`, `tests/test_extension.mjs` — `node tests/<file>` で実行
- `tools/e2e/e2e.mjs` — **公開中のツールを実ブラウザ(ヘッドレスChrome)で操作する結合テスト**。
  `node tools/e2e/e2e.mjs`(1シーンだけなら `E2E_ONLY=payday_slow node tools/e2e/e2e.mjs`)

## テストの守備範囲(2026-07-13。ページを足したら必ず読む)
`tests/*.mjs` が見ているのは `assets/*_core.js` の**純ロジックだけ**で、ページ内の
`<script type="module">`(入力の読み取り・fetchしたデータの適用・結果の描画)は**素通し**になる。
実際にここで実バグを2件出した(下記)。**新しいツール/ページを足したら `tools/e2e/` にシーンを足すこと。**
- **非同期で届く参照データは、待ってから計算する**。`fetch().then(d => DATA = d)` した変数を
  ボタン押下時にそのまま読むと、**回線が遅いユーザーだけ空データで計算される**。
  開発機とCIでは速すぎて100%再現しないので、**E2Eではサーバ側で配信を遅らせて固定する**
  (`payday_slow` シーン)。ボタンは無効化せず `await` で待つ(押せないUIより待つUIの方がよい)
- **データの収録範囲を超えた入力に黙って答えない**。祝日JSONは収録年までしか知らないので、
  範囲外の年は `beyondData` を立てて「概算」と申告する(読み込み失敗時も同じ扱い)。
  `tests/test_payday.mjs` のカナリアが、収録が「今日+6ヶ月」より手前で尽きる状態を落とす

## ドメイン知識(実装済みの仕様)
- 差引方式は3つ: 差引前基準(請求額で手数料判定)/差引後基準(振込額で判定)/一律差引(550円等)
- 3万円閾値は「振込金額」で判定。差引後基準では請求額3万円前後に**解が2つ/0になる不定帯**があり、
  UIで注意書きを出す(calcGoのnotes)。0解のときは高い方の区分を提案
- 銀行別手数料プリセット(fee_table.json)は収集エージェントの検証済みデータが届いてから追加する。
  未検証の手数料をプリセットに入れないこと(誤差引の実害が出る)

## 運用
- ローカル確認: `cd docs && python3 -m http.server 18923`(ESモジュールのためfile://不可)
- デプロイ: 静的ホスティング(Cloudflare Pages / GitHub Pages)。独立収益源のため
  会社インフラとは分離したアカウントを使う(Masahiro判断待ち)
- 収益化: 当面AdSense(サイト全体) → 束ねツールが増えたら個別Pro買い切り(Gumroad)
- 予定: 支払サイトカレンダー、全銀カナ正規化(「不在」の反証検証が通ってから着手)

## サイト構造・記事の型(2026-07-12 全面改善。以後これを守る)
- **トップは「ツール」が主役**: hero → ツールカード(グリッド・アイコン付) → コラム(記事リスト形式)。
  ツールと記事を同じカードで混ぜない(Masahiro指摘)
- ヘッダは全ページ共通ナビ(ブランド + ツール/コラム)
- **記事の必須要素**: パンくず(BreadcrumbList構造化データ) / 公開日・更新日 / 結論ファーストのリード /
  **目次(TOC・h2/h3にid)** / **図解(インラインSVG。外部画像は使わない)** / まとめボックス /
  FAQ / 関連記事 / 出典 / AdSenseスニペット / canonical / sitemap追加
- **FAQ構造化データの設問は、必ず本文にも可視の見出し(h3)として置く**(JSON-LDにだけ書かない)。
  可視コンテンツの無いFAQマークアップはGoogleのガイドライン違反で、リッチリザルト対象外・手動対策の
  対象になりうる。公開前に「JSON-LDのFAQ設問 ⊆ 本文のh3」を必ず確認する(2026-07-13に実際に混入)
- **インラインSVGは座標を手で置くので、必ずヘッドレスChromeで実描画して目で見る**
  (注釈の矢印が意図した対象を指していない等は、描画しないと分からない)
- 記事に**数値の一覧表**を載せるときは、正本JSON(例 `fee_table.json`)から生成し、
  **一致を固定するテストを書く**(例 `tests/test_fee_article.mjs`)。手打ちの転記と、
  データ改定時に記事だけ古い数字が残る事故を防ぐ。本文の断定(「5.1倍差」等)も照合対象に含める
- CSSは docs/assets/style.css に集約(記事内<style>を書かない)。`.note`は警告ボックス用に予約済み、
  見出し脇の補足は`.hint`を使う

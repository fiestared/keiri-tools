# keiri-tools — 経理ミニツールズ（柱A: マイクロWebツール）

AI月30万円プロジェクト・柱Aの初号プロダクト群。経理実務の「ちょっとした計算」を
静的サイト(サーバー処理なし・データ送信なし)で提供し、広告+将来のPro機能で収益化する。
選定根拠: ニッチ調査(gbrain research/jp-micro-webtool-niche-2026-07)で
「振込手数料 先方負担計算機の不在」が唯一敵対的検証を生き残った空白だったため。

## 構成
- `docs/` — 静的サイト一式(そのままデプロイ可能)
  - `index.html` — ハブトップ(ツール一覧)
  - `senpou-futan/` — 振込手数料「先方負担」計算 + 入金差額チェック
  - `assets/senpou_core.js` — 計算純ロジック(DOM非依存)。**変更したら必ずテストを回す**
  - `ext/amazon-receipt/selectors.json` — Chrome拡張のリモートセレクタ定義(下記)。
    AmazonのDOM変更はこのJSONの更新+pushだけで直せる(ストア再審査不要)
- `extension/amazon-receipt/` — アマゾン領収書・電帳法索引簿メーカー(Chrome拡張, MV3)。
  設計・現状・動作確認手順は同フォルダのREADME.md。セレクタは実DOM未検証(フィクスチャ待ち)
- `tests/test_senpou.mjs`, `tests/test_extension.mjs` — `node tests/<file>` で実行
- `tools/e2e/e2e.mjs` — **公開中のツールを実ブラウザ(ヘッドレスChrome)で操作する結合テスト**。

  実行するとシーンごとに `tools/e2e/.chrome-<scene>/` (Chromeのuser-data-dir)が出来る。
  **.gitignore 済み — 絶対にコミットしない**(4千ファイル・69MB。実際に一度入ってしまい2026-07-13に除去)

### 検査を足すときは「落ちるべきものが落ちる」と「通るべきものが通る」を**両方**見る
2026-07-13に**4回**やった(第6便=支払サイトを誤って落とした / 第10便=正しい消費税の注意書きに
誤反応 / 第12便=先方負担を誤って落とした / 第13便=賞与の抽出検算が**正しい表**を落とした)。
**新しい検査が落ちたら、まず疑うのは検査の期待値**。実例:
- 賞与の算出率の表で「扶養人数が増えれば境界額は**必ず上がる**」と書いたが、表には**等しい**境界が
  実在する(4.084%の行で扶養3人も4人も下限300千円)。正しい不変条件は**非減少**
- 「認められない組み合わせを提示していないか」を本文への正規表現で見たら、**否定的な言及**
  (「〜は認められない」という注意書き)にマッチした。**画面は「提示」と「否定」を同じ語で含む**
落とすべきものが落ちることだけ確かめて満足すると、**正しい商品を壊す修正**に進む。

### 集合一致の網を張ったら、**網の外に何が残るか**を必ず数える（第24・25便／2026-07-14 第1便で3例目）
金額を「カンマ区切りの数の**集合一致**」で見るのは強いが、**同じ量が別の表記で書かれていると網に入らない**。
- 第24便: **等級**（22/25）はカンマを含まず、集合に最初から入っていないので壊しても緑だった
- 第25便: **「32万円」**（万円表記）も同じ。しかも `text.includes("32万円")` で補っていたため、
  **1か所を35万円に壊しても他の箇所に32万円が残って緑**だった（＝存在確認の再発。上の規則を自分で破った）
- 07-14: **日数**（14日・28日・180日）。育休記事で要件の「**14日以上**」を「10日以上」に壊しても緑だった。
  **14日は制度の要件そのもの**で、間違えれば読者に実害が出る
→ **表記の系統ごとに集合を張る**（`/\d{1,3}(,\d{3})+/`・`/\d+万円/`・`/\d+%/`・`/\d+日/` は**別の網**）。
  集合に入らない主張（条文番号・引用の核心）は**載っている要素を名指し**する。
  新しい検査を書いたら「**この網からこぼれる主張は何か**」を口に出して数えること。
  **日数の網は日付を先に除去する**（「2026年8月1日」の"1日"を日数と誤認し、**正しい記事を落とす**）。

### 壊しテストの**壊し方**も一意でなければならない（2026-07-14 第2便）
壊しテストで「素通し」に見えたら、**検査が弱いのか、壊し方が別の場所に当たったのか**を区別する。
実例: 「99条引用の準用指示を削除」で壊したつもりが、**同じ文字列がFAQのJSON-LD（生成物）にも先に出る**ため
`replace(a, b, 1)` はそちらを書き換え、**本文のblockquoteは無傷**だった（＝検査は正しく、壊し方が外れていた）。
→ 壊し方は**タグごと指定**して本文の該当要素を狙う。素通しを見たら、まず壊れているかを確かめる。

### 要素の名指しは、**その名前が一意でなければ効かない**（2026-07-14。「位置で見る」の一歩先）
表の行を「`母親` を含む行」で特定したら、**父親の行が当たった** — 父親の行は配偶者の欄に
「母親（産後休業中）」と書いてあるため。`find()` は**最初の一致**を返すので、
**父親の行を母親の行として検査していた**（＝非対称の反転を検出できない）。
→ **主語のセルで特定する**（`<td><b>母親</b>が受け取る場合</td>`）。要素を名指しするだけでは足りない。

### `<title>` と `<meta name="description">` も検査対象に入れる（2026-07-14）
タグ剥がし（`replace(/<[^>]+>/g," ")`）は**属性値ごと消す**ので、meta descriptionの数字は
**本文だけを見る検査から漏れる**（上限額を書き換えても緑だった）。
**meta descriptionは検索結果に出る＝公開された主張**。title と併せて `text` に連結して数える。

### `git add -A` は「未追跡にしてある」という意図を知らない（2026-07-14）
**意図的に未追跡にしていた重複記事**（`.nopublish` 付き）を `git add -A` が巻き込み、**本番にpushした**。
一覧・sitemapには載らないが**URLは生きる**（重複記事は検索で互いを食い合う）。
→ 記事のコミットは**足すファイルを明示**するか、`git status` を見てから add する。
なお **`rm` は許可外だが `git rm` は使える**（作業ツリーのゴミもこれで消せる）。

`tests/*.mjs` が見ているのは `assets/*_core.js` の**純ロジックだけ**で、ページ内の
`<script type="module">`(入力の読み取り・fetchしたデータの適用・結果の描画)は**素通し**になる。
実際にここで実バグを2件出した(下記)。**新しいツール/ページを足したら `tools/e2e/` にシーンを足すこと。**
> これも散文では守られなかった(2026-07-13 第14便)。**需要が最大の看板ツール(社会保険料)だけ
> E2Eシーンが1つも無く**、「料率は届いているのに『読み込めませんでした』と言い続ける」**全損**を
> 本番で放置していた。単体テスト(`test_shaho.mjs`/`test_shaho_oracle.mjs`)は**全緑のまま**だった
> — coreは正しく、壊れていたのはページだったため。→ `tools/e2e/e2e.mjs` の**網羅チェック**で機械化:
> `assets/*_core.js` を読み込む全ページに、**正常条件で正しい答えを出すシーン**を必ず1つ要求する。
> **404・遅延の再現シーンは網羅に数えない**(壊れたツールでも通るため。実際 `shaho_nodata` は
> 「常に読み込み失敗と言う」全損状態で緑だった)

### 「答えに要るもの」と「飾り」を同じ catch に入れない(2026-07-13 第14便)
参照データの取得を `.catch(() => false)` で包み、失敗したら答えない(fail closed)のは正しい。
だが**その then の中に飾りの描画まで書くと、飾りのバグがデータ取得の失敗に化ける**。
社会保険料ツールは出典ノートの参照キーを間違えており(`_meta.applies_from`。正しくは
`rate_applies_from`)、その `TypeError` を catch が拾って「料率データを読み込めませんでした。
**通信環境を確認して再読み込みしてください**」と言い続けた。料率は正常に届いており、
**都道府県プルダウンも料率入りで描けている**ので画面は正常に見え、利用者は永久に無意味な
再読み込みを促される。→ catch が拾ってよいのは**データが手に入らなかったこと**だけ。
飾り(出典・年度の表示)は `ratesReady.then(...)` で**別に**描き、そこで転んでも計算は止めない。

### 年度・年分は**データに持たせて画面に出す**。ページに手書きしない
「令和8年度」をページに書くと、料率JSONを差し替えた年に**画面だけが古い年度を名乗る**(嘘をつく)。
参照データの `_meta.year`(社保) / `year`(税額表)を正本にし、結果と出典表示はそこから描く。
> title・meta・JSON-LDはSEOのため静的HTMLに書くしかないので、**`node tests/test_year_staleness.mjs`
> が「データの年 ≠ ページの手書き」を落とす**。★**一括置換するな**: 「令和8年分の表で計算した」
> (=データの申告。差し替えたら嘘)と「支援金は令和8年4月に新設」(=**制度の事実**。何年経っても真)は
> 別物で、後者を置換すると**歴史を書き換えて嘘にする**。事実は `HISTORICAL_FACTS` に理由つきで免除する。
- **非同期で届く参照データは、待ってから計算する**。`fetch().then(d => DATA = d)` した変数を
  ボタン押下時にそのまま読むと、**回線が遅いユーザーだけ空データで計算される**。
  開発機とCIでは速すぎて100%再現しないので、**E2Eではサーバ側で配信を遅らせて固定する**
  (`payday_slow` シーン)。ボタンは無効化せず `await` で待つ(押せないUIより待つUIの方がよい)
- **データの収録範囲を超えた入力に黙って答えない**。祝日JSONは収録年までしか知らないので、
  範囲外の年は `beyondData` を立てて「概算」と申告する(読み込み失敗時も同じ扱い)。
  `tests/test_payday.mjs` のカナリアが、収録が「今日+6ヶ月」より手前で尽きる状態を落とす
- **日付の計算・比較をページ側に書かない**(2026-07-13 有給ツールで実バグ3件。全て`*_core.js`へ移した)。
  ページに日付計算があると単体テストが1行も届かない。以下は経理ツールでは全て**実害**になる:
  - `new Date(y, m + N, d)` は**存在しない日を翌月へ繰り越す**。`2/31` → `3/3`。
    月末入社の有給付与日が法定より遅くなった。**Nヶ月後は「1日」で月だけ出し、日は末日でクランプ**する
    (民法143条2項: 応当する日がない月はその月の末日)。`addMonthsClamped()` を使う
  - `new Date("YYYY-MM-DD")` は**UTCとして解釈される**。JST(+9)では当日の 00:00〜09:00 が
    「まだ来ていない」と判定される。**日付どうしは `YYYY-MM-DD` の文字列比較**で行う(TZ非依存)
  - **画面の見出しと表は同じ1つの計算結果から描く**。別々に日付を計算すると、
    「10日付与済み」と「初回付与は未来(予定)」を**同時に表示**する(実際に出た)
- E2Eの入社日・基準日は**過去の固定日**にする。「今日」に依存するシーンは実行日によって落ちる/通る

## ドメイン知識(実装済みの仕様)
- 差引方式は3つ: 差引前基準(請求額で手数料判定)/差引後基準(振込額で判定)/一律差引(550円等)
- 3万円閾値は「振込金額」で判定。差引後基準では請求額3万円前後に**解が2つ/0になる不定帯**があり、
  UIで注意書きを出す(calcGoのnotes)。0解のときは高い方の区分を提案
- 銀行別手数料プリセット(fee_table.json)は収集エージェントの検証済みデータが届いてから追加する。
  未検証の手数料をプリセットに入れないこと(誤差引の実害が出る)
- **全銀の使用可能文字には2階層ある(2026-07-13に混同しかけた。正本: gbrain tools/zengin-kana-normalization-research)**:
  - **レコード全体の文字集合**(銀行配布の「全銀仕様データレコード使用可能文字」表): `( ) ｢ ｣ / - . ¥` を含む
  - **フィールド別の制限** ← 受取人名・口座名・振込依頼人名はこちら。
    全銀協「使用文字一覧」注1が「記号**4種類 `( ) - .` のみ**」と明記。`/ ¥ ｢ ｣ ,` は**使えない**
  - 銀行のPDFを見て許可集合を広げたくなったら、それがどちらの階層かを必ず確かめる
  - カナは **ヲ・小文字・長音ｰ が不可**。長音ー→`-`、中黒・→`.`、小書き→大文字。
    **ヰ/ヱ/ヵ/ヶ は半角に存在しない**ので イ/エ/カ/ケ へ倒す(ヱビス・ヰセキは実在の名義)

## 記事を書くとき(2026-07-13 量産体制。**まず `tools/ARTICLE_SPEC.md` を読む**)
記事の型の正本は **`tools/ARTICLE_SPEC.md`**。下の「記事の必須要素」は、いまは**機械が強制する**:
- `node tests/test_article_structure.mjs` — 型の違反を落とす(**書いた瞬間に既存9記事から14件出た**。
  散文の約束は守られない、の実例。canonical/GA4/AdSense/構造化データ/目次と全h2の対応/
  **図解(figure内インラインSVG)**/FAQ/出典/ツール導線/title 60字/**sitemap・一覧への掲載**)
- `node tools/gen_index_sitemap.mjs` — **sitemap.xml / コラム一覧 / トップの新着6本を生成**。
  手で足さない(足し忘れた記事は誰にも届かない。トップには**古い低需要の記事が居座っていた**)
- `node tools/gen_faq_jsonld.mjs` — FAQのJSON-LDを本文から生成
- `python3 tools/keyword_demand.py <語>` — **書く前に需要を測る。月1,000検索未満は書かない**

**⚠️ 政府サイトの数値確認に WebFetch を使うな。要約器が「もっともらしい嘘」を返す。**
実例2件(2026-07-13): ①国税庁の2割特例のページで、対象者を「認定経営革新等支援機関の支援を
受けた小規模事業者」と**完全に捏造**した。②協会けんぽの料率ページで「最低は沖縄県9.44%」と回答
(正しくは新潟県9.21%。**同じ回答の中に9.21%を列挙しておきながら矛盾していた**)。
**エラーにならないので気づけない。** → **必ず `curl` で生テキストを読む。**
- **WebSearchの「検索結果の要約」も同じ**(第19便で3例目)。URLを見つける道具としてだけ使い、
  **中身は信じない**。実例: 公取委リーフレットの「合意の上で委託事業者が補填すれば問題ない」を
  **振込手数料の例外**として要約してきたが、生PDFではその※は**隣の論点(受取手数料=決済手段の手数料)**に
  付いた注で、**振込手数料に補填の例外は無い**。要約器は**隣接する2つの論点を混ぜる**
- **nta.go.jp の文字コードは、ディレクトリで決め打つな。ページごとに `meta charset` を見る**
  (2026-07-13 第21便。第17便で「taxanswer=UTF-8 / publication=Shift_JIS」と**ディレクトリ単位で
  規則化したのが誤り**だった)。**同じディレクトリ内でも混在する**:
  `taxanswer/shotoku/1130.htm` は **UTF-8** なのに、その質疑応答 `1130_qa.htm` は **Shift_JIS**。
  ```
  curl -s <URL> | grep -io 'charset=[a-z0-9_-]*' | head -1    # ← 読む前にこれ
  ```
  UTF-8のつもりで読むと**全文が文字化けする**。文字化けを「このページには情報が無い」と誤読すると、
  **記事の核心が入った質疑応答を丸ごと落とす**(第21便で実際に落としかけた)
- **e-Gov法令APIは v2 を使う**(`/api/2/law_data/{law_id}`)。**v1は改正前の古い条文を返す**
  (措法67条の5で「30万円/令和8年3月31日」＝改正前が返った)
- **未施行リビジョンに注意**。令和8年度改正の所得税法は**施行日が令和8年12月1日**なので、
  今日の「現行」を引くと**改正前の額が返る**。`/api/2/law_revisions/{law_id}` で将来施行版
  (`..._20261201_508AC0000000012`)を読む。**踏み外すと「正しい記事」を誤りと判定する**
- PDFは `pdftotext -layout`。**表は結合セルで行がずれる**ので `pdfplumber` でbboxごと取る
- **改正直後は国税庁のタックスアンサーが古いままのことがある。財務省の大綱が正本**
正本: gbrain `research/nta-site-webfetch-hallucination`

**並列で記事を書かせるときの規律**: 各エージェントは**自分の記事1ファイルだけ**を書く。

## サイト構造・記事の型(2026-07-12 全面改善。以後これを守る)
- **トップは「ツール」が主役**: hero → ツールカード(グリッド・アイコン付) → コラム(記事リスト形式)。
  ツールと記事を同じカードで混ぜない(Masahiro指摘)
- ヘッダは全ページ共通ナビ(ブランド + ツール/コラム)
- **記事の必須要素**: パンくず(BreadcrumbList構造化データ) / 公開日・更新日 / 結論ファーストのリード /
  **目次(TOC・h2/h3にid)** / **図解(インラインSVG。外部画像は使わない)** / まとめボックス /
  FAQ / 関連記事 / 出典 / AdSenseスニペット / canonical / sitemap追加
- **FAQ構造化データ(JSON-LD)は手で書かない。本文から生成する** — `node tools/gen_faq_jsonld.mjs`。
  人が触るのは本文の「よくある質問」ブロックだけ(`<h2>よくある質問</h2>` または `<h2 data-faq>` に続く
  `<h3>設問</h3><p>答え</p>` の並び。答えは**h3直後の`<p>`ひとつ**。表やcalloutはFAQに入れない
  — Googleが `acceptedAnswer` に許可するタグに table は無い)。
  Googleの要件は「**設問と回答の全文**が利用者に見えていること」。`tests/test_faq_visible.mjs` が
  設問・答えの両方の可視性と一致を機械で強制する(独立実装で照合。生成器をimportしない)。
  > 経緯: 2026-07-13 第1便に「以後 設問⊆h3 を確認する」と**散文で**書いたが機械化せず、第8便に
  > 11ページ31件の違反が発覚。しかも見ていたのは設問だけで、**答えの全文はどのページにも無かった**
  > (JSON-LDが、ページに存在しない文章を構造化データとして申告していた)。第9便で生成方式にして解消。
  > **手で2箇所を同期し続ける設計は必ず腐る。片方を生成物にする**
- **インラインSVGは座標を手で置くので、必ずヘッドレスChromeで実描画して目で見る**
  (注釈の矢印が意図した対象を指していない等は、描画しないと分からない)
- 記事に**数値の一覧表**を載せるときは、正本JSON(例 `fee_table.json`)から生成し、
  **一致を固定するテストを書く**(例 `tests/test_fee_article.mjs`)。手打ちの転記と、
  データ改定時に記事だけ古い数字が残る事故を防ぐ。本文の断定(「5.1倍差」等)も照合対象に含める
- CSSは docs/assets/style.css に集約(記事内<style>を書かない)。`.note`は警告ボックス用に予約済み、
  見出し脇の補足は`.hint`を使う

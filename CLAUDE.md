# keiri-tools — 経理ミニツールズ（柱A: マイクロWebツール）

AI月30万円プロジェクト・柱Aのプロダクト。経理実務の「ちょっとした計算」を
静的サイト(サーバー処理なし・データ送信なし)で提供し、広告+将来のPro機能で収益化する。
現在: ツール5本(社会保険料/源泉徴収/営業日/有給/消費税) + コラム記事。
**足りないのは入口(記事)** — ツールに検索需要があっても、記事がないと辿り着かれない。

## 構成
- `docs/` — 静的サイト一式(そのままデプロイ可能)。`assets/*_core.js` = 計算純ロジック(DOM非依存)
- `docs/ext/amazon-receipt/selectors.json` — Chrome拡張のリモートセレクタ定義。
  AmazonのDOM変更はこのJSONの更新+pushだけで直せる(ストア再審査不要)
- `extension/amazon-receipt/` — アマゾン領収書・電帳法索引簿メーカー(Chrome拡張, MV3)。詳細は同フォルダのREADME.md
- `tests/*.mjs` — `node tests/<file>` で実行
- `tools/e2e/e2e.mjs` — **公開中のツールを実ブラウザ(ヘッドレスChrome)で操作する結合テスト**。
  `node tools/e2e/e2e.mjs` (1シーンだけなら `E2E_ONLY=payday_slow node tools/e2e/e2e.mjs`)。
  実行すると `tools/e2e/.chrome-<scene>/` が出来る。**.gitignore済み — 絶対にコミットしない**(4千ファイル・69MB)

## 記事を書くときの入口
**型の正本は `tools/ARTICLE_SPEC.md`。書く前に必ず読む。**
- `python3 tools/keyword_demand.py <語>` — **書く前に需要を測る。月1,000検索未満は書かない**
- `node tests/test_article_structure.mjs` — 型の違反を落とす(canonical/GA4/AdSense/構造化データ/
  目次と全h2の対応/図解(figure内インラインSVG)/FAQ/出典/ツール導線/title 60字/一覧・sitemap掲載/**ORDER登録**)
- `node tools/gen_index_sitemap.mjs` — **sitemap.xml / コラム一覧 / トップの新着6本を生成**(手で足さない)
- `node tools/gen_faq_jsonld.mjs` — FAQのJSON-LDを**本文から生成**(手で書かない)

**並列で記事を書かせるときの規律**: 各エージェントは**自分の記事1ファイルだけ**を書く。
`sitemap.xml` / `column/index.html` / `assets/style.css` は**中央(親)が更新**する。
`gen_faq_jsonld.mjs` は全ページを書き換えるので、**並列中に子に流させない**(他の作業中ファイルを壊す)。

# 検査(テスト)の9つの規則

> **なぜこれだけ規則があるのか**: この9つは全て**「検査が緑なのに商品が間違っている」**＝黙って間違える、
> の別の顔。**道具を強くするたびに、その道具の外側に新しい空白が生まれる**ので増えてきた。
> **経緯・実例・失敗の全記録は gbrain `ops/webtool-silent-wrong-answer-patterns`**(§17〜§23が記事編)。
> 規則の意味が分からなくなったらそちらを読む。以下は**実行時に必要な規則だけ**。
>
> **📌 次に自戒を得たときの書き足し先**(2026-07-14 第8便に27.8KB→21KBへ痩せさせた。再肥大を防ぐ規律):
> **経緯・実例・便番号は必ず gbrain へ書く。CLAUDE.md は触らない**のが既定。
> ここに規則を**足してよいのは、上の9つのどれにも当てはまらない新種**のときだけ
> (既存規則の再発なら、gbrain に実例を足すだけでよい — **規則は既にここに在るのに破ったのだから、
> 同じことをもう一度書いても防げない**)。常時読み込まれる層が太ると全体の精度が落ちる。

### 1. 「落ちるべきものが落ちる」と「通るべきものが通る」を**両方**見る
**新しい検査が落ちたら、まず疑うのは検査の期待値**(2026-07-13だけで4回、記事でも2回、私の側が誤りだった)。
落ちるべきものが落ちることだけ確かめて満足すると、**正しい商品を壊す修正**に進む。
- 例: 「扶養が増えれば境界額は必ず上がる」と書いたが表には**等しい**境界が実在した(正しくは非減少)
- 例: タグ剥がしが `= <b>5</b>` を `=  5`(空白2つ)にするので `"= 5"` が不一致になった(記事は正しかった)

### 2. 壊しテストは、壊す前に**「無傷が緑」を確かめる**（ベースライン確認）
検査が**常に赤**なら**何を壊しても赤**になり、壊しテストは「22/22捕捉」と**嘘の満点**を出す。
実際に起き、**検証していない記事を「検証済み」と称して公開する**ところだった。
→ 壊しテストの先頭で本物の記事に検査を1回走らせ、**緑でなければ即座に降りる**。
**規則1は壊しテスト自身にも要る。**

### 3. 「本文のどこかに在る」で見るな。**その主張が載っている要素を名指し**する
語の**存在**を本文全体への正規表現で確かめる検査は、**同じ語が別の場所にもあると素通しする**。
**記事のFAQ・出典は本文の主張を必ず言い換えて再掲する**ので、
**本文一致の検査は構造的にほぼ必ず素通しする**。
→ **記事の検査は、原則すべて要素(callout/blockquote/表の行)を名指しして書く**。
（この規則の再発は通算18回。**規則を知っている状態でも破る**ので、壊しテストで殴って確かめること）

### 4. 名指しは**一意**でなければ効かない
「`母親` を含む行」で特定したら**父親の行が当たった**(父親の行は配偶者欄に「母親(産後休業中)」と書いてある)。
`find()` は**最初の一致**を返す。→ **主語のセルで特定する**(`<td><b>母親</b>が受け取る場合</td>`)。

### 5. 名指しの**粒度**まで下ろす — 名指しした要素が、自分の中で主張を再掲する
`<div class="callout">` を名指ししても、**見出しの`<b>`が中の`<p>`と同じ主張を含んでいたら**、
本文を消しても緑になる。**callout / 表 / blockquote は単位として大きすぎることがある**。
→ **名指しは「その主張が1回しか現れない最小の要素」まで下ろす**。

### 6. 集合一致の網を張ったら、**網の外に何が残るか**を数える
金額を「カンマ区切りの数の集合一致」で見るのは強いが、**同じ量が別の表記だと網に入らない**。
→ **表記の系統ごとに別の網を張る**: `/\d{1,3}(,\d{3})+/`・`/\d+万円/`・`/\d+%/`・`/\d+日/`・`/\d+倍/`。
- **等級(22/25)・3桁の数(736円・310円)はカンマを含まず、金額の網に構造上入らない**
- **日数の網は日付を先に除去する**(「2026年8月1日」の"1日"を日数と誤認し**正しい記事を落とす**)。
  日付は**年の付かない形も剥がす**(「前年9月30日時点」の"30日")
- 集合に入らない主張(条文番号・引用の核心)は**要素を名指し**する

**新しい検査を書いたら「この網からこぼれる主張は何か」を口に出して数えること。**

### 7. 網は「値の過不足」に強いが、**同じ値が複数箇所に出る主張の位置ずれ**には無力
納付期限「その月の**10日**」を**本文だけ20日に壊しても、日数の網は緑**だった
(同じ"10日"がFAQの答えと出典にもあるので集合には10が残る)。
網は「記事のどこかに10日がある」しか見ておらず、**その主張の場所が正しいか**を見ていない。
→ **2箇所以上に出る数字の主張は、網に加えて要素の名指しが要る**(網は「安心」を与えるが守っていない)。

### 8. 壊しテストの**壊し方**も一意でなければならない
**素通しを見たら「検査が弱いのか、壊し方が外れたのか」を区別する**(この区別が何度も効いている)。
実例: 狙った文字列が**headのFAQ JSON-LDに先に出る**ため `replace(a,b,1)` がそちらに当たり、
**本文は無傷**だった(＝検査は正しく、壊し方が外れていた)。→ 壊し方は**タグごと指定**して本文の要素を狙う。

### 9. `<title>` と `<meta name="description">` も検査対象に入れる
タグ剥がし(`replace(/<[^>]+>/g," ")`)は**属性値ごと消す**ので、meta descriptionの数字は
**本文だけを見る検査から漏れる**。**meta descriptionは検索結果に出る＝公開された主張**。

# 実装の規則（黙って間違えないための作り）

## 単体テストの守備範囲 — ページ内スクリプトは**素通し**する
`tests/*.mjs` が見ているのは `assets/*_core.js` の**純ロジックだけ**。ページ内の
`<script type="module">`(入力の読み取り・fetchしたデータの適用・結果の描画)は検査されない。
実際に**看板ツールの全損**(料率は届いているのに「読み込めませんでした」と言い続ける)を本番で7便放置した
— coreは正しく、壊れていたのはページだったので**単体は全緑のまま**だった。

→ **新しいツール/ページを足したら `tools/e2e/` にシーンを足す**。
`e2e.mjs` の**網羅チェック**が機械で強制する: `assets/*_core.js` を読み込む全ページに
**「正常条件で正しい答えを出すシーン」を必ず1つ**要求する。
**404・遅延の再現シーンは網羅に数えない**(壊れたツールでも通るため)。
E2Eの入社日・基準日は**過去の固定日**にする(「今日」に依存するシーンは実行日によって落ちる/通る)。

## 参照データを使うページの3つの義務（`node tests/test_data_pages.mjs` が強制）
**assets配下のJSONをfetchする全ページ**に「**待つ / 読み込み失敗を申告 / 収録範囲外を申告**」を強制する。
新しいツールを足すと落ちるので、落ちたらこの節を読む。

- **検査対象をデータファイル名で絞らないこと**。名前で絞ると**別の参照データを足したときに新ページが漏れる**
  (実際に同じ失敗モードの既存2ページが7便のあいだ見えていなかった)。
  **「非同期の参照データを使っている」という形で拾う**
- 「待つ」規律が要るのは**そのデータが答えの一部になるとき**だけ。表示専用のデータ(入力欄を埋める候補)に
  強制すると**正しい商品を落とす検査**になるので `PRESENTATION_ONLY` に**理由つきで**書いて免除する
  (ただし**読み込み失敗の申告は表示専用でも要る**)
- **非同期で届く参照データは、待ってから計算する**。`fetch().then(d => DATA = d)` した変数をボタン押下時に
  そのまま読むと、**回線が遅いユーザーだけ空データで計算される**。開発機とCIでは速すぎて再現しないので、
  **E2Eではサーバ側で配信を遅らせて固定する**(`payday_slow`)。ボタンは無効化せず `await` で待つ
- **データの収録範囲を超えた入力に黙って答えない**。範囲外は `beyondData` を立てて「概算」と申告する
  (読み込み失敗時も同じ扱い)。`tests/test_payday.mjs` のカナリアが、収録が「今日+6ヶ月」より手前で
  尽きる状態を落とす

## 「答えに要るもの」と「飾り」を同じ catch に入れない
参照データの取得を `.catch(() => false)` で包み、失敗したら答えない(fail closed)のは正しい。
だが**その then の中に飾りの描画まで書くと、飾りのバグがデータ取得の失敗に化ける**。
実例: 出典ノートの参照キー誤り(`_meta.applies_from`)の `TypeError` を catch が拾い、
料率は正常に届いているのに「**通信環境を確認して再読み込みしてください**」と言い続けた
(画面は正常に見えるので利用者は永久に無意味な再読み込みを促される)。

→ **catch が拾ってよいのは「データが手に入らなかったこと」だけ**。
飾り(出典・年度の表示)は `ratesReady.then(...)` で**別に**描き、そこで転んでも計算は止めない。

## 年度・年分は**データに持たせて画面に出す**。ページに手書きしない
「令和8年度」をページに書くと、料率JSONを差し替えた年に**画面だけが古い年度を名乗る**(嘘をつく)。
参照データの `_meta.year`(社保) / `year`(税額表)を正本にし、結果と出典表示はそこから描く。
title・meta・JSON-LDはSEOのため静的HTMLに書くしかないので、
**`node tests/test_year_staleness.mjs` が「データの年 ≠ ページの手書き」を落とす**。

★**一括置換するな**: 「令和8年分の表で計算した」(=**データの申告**。差し替えたら嘘)と
「支援金は令和8年4月に新設」(=**制度の事実**。何年経っても真)は別物で、
後者を置換すると**歴史を書き換えて嘘にする**。事実は `HISTORICAL_FACTS` に**理由つきで**免除する。

## 日付の計算・比較を**ページ側に書かない**(全て `*_core.js` へ)
ページに日付計算があると単体テストが1行も届かない。以下は経理ツールでは全て**実害**になる:
- `new Date(y, m + N, d)` は**存在しない日を翌月へ繰り越す**(`2/31`→`3/3`)。月末入社の有給付与日が
  法定より遅くなった。→ **Nヶ月後は「1日」で月だけ出し、日は末日でクランプ**する(民法143条2項)。
  `addMonthsClamped()` を使う
- `new Date("YYYY-MM-DD")` は**UTCとして解釈される**。JST(+9)では当日の00:00〜09:00が「まだ来ていない」と
  判定される。→ **日付どうしは `YYYY-MM-DD` の文字列比較**で行う(TZ非依存)
- **画面の見出しと表は同じ1つの計算結果から描く**。別々に日付を計算すると「10日付与済み」と
  「初回付与は未来(予定)」を**同時に表示**する(実際に出た)

## 記事の数値と実装の照合
記事に**数値の一覧表**を載せるときは、正本JSON(例 `fee_table.json`)から生成し、**一致を固定するテストを書く**
(例 `tests/test_fee_article.mjs`)。本文の断定(「5.1倍差」等)も照合対象に含める。
手打ちの転記と、データ改定時に**記事だけ古い数字が残る**事故を防ぐ。

# 一次情報の読み方（数字を書く前に必ず）

## ⚠️ 政府サイトの数値確認に **WebFetch を使うな**。要約器が「もっともらしい嘘」を返す
エラーにならないので気づけない。**必ず `curl` で生テキストを読む。**
- 実例: 国税庁の2割特例で対象者を**完全に捏造**した / 協会けんぽの料率で「最低は沖縄県9.44%」と回答
  (正しくは新潟県9.21%。**同じ回答の中に9.21%を列挙しておきながら矛盾していた**)
- **WebSearchの「検索結果の要約」も同じ**。URLを見つける道具としてだけ使い、**中身は信じない**。
  要約器は**隣接する2つの論点を混ぜる**(公取委リーフレットで、隣の論点に付いた注を振込手数料の例外として要約した)

正本: gbrain `research/nta-site-webfetch-hallucination`

## 取得のしかた
- **nta.go.jp の文字コードは、ディレクトリで決め打つな。ページごとに `meta charset` を見る**
  (同じディレクトリ内でも混在する: `taxanswer/shotoku/1130.htm`はUTF-8、その質疑応答`1130_qa.htm`はShift_JIS)。
  ```
  curl -s <URL> | grep -io 'charset=[a-z0-9_-]*' | head -1    # ← 読む前にこれ
  ```
  UTF-8のつもりで読むと**全文が文字化けする**。文字化けを「情報が無い」と誤読すると**記事の核心を丸ごと落とす**
- **e-Gov法令APIは v2 を使う**(`/api/2/law_data/{law_id}`)。**v1は改正前の古い条文を返す**
- **未施行リビジョンに注意**。令和8年度改正の所得税法は**施行日が令和8年12月1日**なので、今日の「現行」を
  引くと**改正前の額が返る**。`/api/2/law_revisions/{law_id}` で将来施行版を読む。
  **踏み外すと「正しい記事」を誤りと判定する**
- PDFは `pdftotext -layout`。**表は結合セルで行がずれる**ので `pdfplumber` でbboxごと取る
- **改正直後は国税庁のタックスアンサーが古いままのことがある。財務省の大綱が正本**

## 記事の目玉は「外部オラクル」で裏取りする
記事の核心となる主張は、**自分の算数ではなく一次情報の公表値**で照合する
(例: 協会けんぽ公式例 17万→5,670→3,780円を実装が1円まで再現する / 厚労省の公表2値だけから
3業種すべての公表労働者負担が再現する)。**別々の資料の数字が噛み合えば、読み違えていないと分かる。**

# サイト構造・記事の型

- **トップは「ツール」が主役**: hero → ツールカード(グリッド・アイコン付) → コラム(記事リスト形式)。
  ツールと記事を同じカードで混ぜない(Masahiro指摘)
- ヘッダは全ページ共通ナビ(ブランド + ツール/コラム)
- **記事の必須要素**: パンくず(BreadcrumbList) / 公開日・更新日 / 結論ファーストのリード /
  目次(TOC・h2/h3にid) / **図解(インラインSVG。外部画像は使わない)** / まとめボックス /
  FAQ / 関連記事 / 出典 / AdSenseスニペット / canonical / sitemap・一覧・ORDERへの登録
- **FAQ構造化データ(JSON-LD)は手で書かない。本文から生成する** — `node tools/gen_faq_jsonld.mjs`。
  人が触るのは本文の「よくある質問」ブロックだけ(`<h2>よくある質問</h2>` に続く `<h3>設問</h3><p>答え</p>`。
  答えは**h3直後の`<p>`ひとつ**。表やcalloutはFAQに入れない — Googleが `acceptedAnswer` に許可するタグに
  table は無い)。Googleの要件は「**設問と回答の全文**が利用者に見えていること」で、
  `tests/test_faq_visible.mjs` が可視性と一致を機械で強制する(独立実装で照合。生成器をimportしない)
- **インラインSVGは座標を手で置くので、必ずヘッドレスChromeで実描画して目で見る**
  (注釈の矢印が意図した対象を指していないのは、描かないと分からない。実際に「宙に浮いた線」を出した)
- CSSは `docs/assets/style.css` に集約(記事内`<style>`を書かない)。`.note`は警告ボックス用に予約済み、
  見出し脇の補足は`.hint`

> **手で2箇所を同期し続ける設計は必ず腐る。片方を生成物にする。**
> ただし**生成物にしただけでは足りない**: 生成器の入力が手管理(`ORDER`配列)なら腐るし、
> 更新専用の生成器は**黙って何もしない**。生成器は**gitの追跡状態を知らない**ので作業ツリーのゴミも本番に混ぜる。

# ドメイン知識(実装済みの仕様)
- 差引方式は3つ: 差引前基準(請求額で手数料判定)/差引後基準(振込額で判定)/一律差引(550円等)
- 3万円閾値は「振込金額」で判定。差引後基準では請求額3万円前後に**解が2つ/0になる不定帯**があり、
  UIで注意書きを出す(calcGoのnotes)。0解のときは高い方の区分を提案
- 銀行別手数料プリセット(fee_table.json)は**検証済みデータが届いてから**追加する。
  未検証の手数料をプリセットに入れないこと(誤差引の実害が出る)
- **全銀の使用可能文字には2階層ある**(正本: gbrain `tools/zengin-kana-normalization-research`):
  - **レコード全体の文字集合**(銀行配布の「使用可能文字」表): `( ) ｢ ｣ / - . ¥` を含む
  - **フィールド別の制限** ← 受取人名・口座名・振込依頼人名はこちら。全銀協「使用文字一覧」注1が
    「記号**4種類 `( ) - .` のみ**」と明記。`/ ¥ ｢ ｣ ,` は**使えない**
  - 銀行のPDFを見て許可集合を広げたくなったら、**それがどちらの階層か**を必ず確かめる
  - カナは **ヲ・小文字・長音ｰ が不可**。長音ー→`-`、中黒・→`.`、小書き→大文字。
    **ヰ/ヱ/ヵ/ヶ は半角に存在しない**ので イ/エ/カ/ケ へ倒す(ヱビス・ヰセキは実在の名義)

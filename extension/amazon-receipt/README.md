# アマゾン領収書・電帳法索引簿メーカー (Chrome拡張)

Amazon.co.jpの注文履歴から**電子帳簿保存法(電子取引)の索引簿CSV**を生成する拡張。
検索要件の3要素(取引年月日・取引金額・取引先)を必ず含み、ExcelでそのままJSON崩れなく
開ける(UTF-8 BOM+CRLF)。処理は全てブラウザ内で完結し、注文データは外部送信しない
(外部通信はセレクタ定義JSONの取得のみで、ユーザーデータは含まない)。

## 全ページ巡回(Pro)の設計 — なぜ「ページを移動する」のか

**裏からfetchして次ページを読む方式は、Amazonに通用しない**(v0.3までの実装。2026-07-14に実測):

| やり方 | 結果 |
|---|---|
| 1ページ目(実DOM) | 正しく10件 |
| 2ページ目を `fetch` | 正しく10件(新規) |
| **3ページ目を `fetch`**(`?startIndex=20&ref_=..._2_3`) | 10件読めるが**全部が既出**(＝別ページの中身が返る) |
| 同じURLを利用者が**ブラウザで開く** | ちゃんと3ページ目が出る |
| `?startIndex=20`(**ref_なし**)を開く | **1ページ目に戻る** |

→ 裏取得はAmazonの気まぐれに依存していて信用できない。
**利用者がクリックするのと同じ経路**、つまり `location.href` で実際にページを移動しながら集める。

content script は毎ページのロードで走るので、状態を `chrome.storage.local`(`kt_crawl`)に持ち回す:

```
{ active, startedAt, runId, maxPages, pages, visited: [正規化URL...], orders: [...] }
```

1ページごとに **読む → 積む(注文番号で重複除去) → 次へ移動 or 完了**。
判断はすべて純ロジック **`src/lib/crawl.js`** にあり、`tests/test_extension_crawl.mjs` が守る。

**必ず止まるための番人(どれ1つ欠けても無限ループか重複CSVになる)**:

- **新規0件で必ず打ち切る** — Amazonは**範囲外のページで1ページ目を返す**ので、これが最終防衛線
- **訪問済みのURLへは絶対に移動しない** — 比較は `ref_` を落として正規化する
  (`ref_` は同じページでも文脈で変わるので、生URLでは同一と気づけない)
- **maxPages(既定30)で打ち切る**
- **10分以上前の「巡回中」は破棄する** — 前回の中断が残っていて、次に注文履歴を開いたら
  **勝手にページが移動し始める**、という事故を防ぐ
- **「次へ」リンクからしか次ページURLを取らない** — `startIndex` から組み立てたURL(ref_なし)へ
  移動すると1ページ目に戻ってしまう(`ktNextPageUrlByIndex` を移動先に使ってはいけない)
- **巡回を始めたタブでしか再開しない**(`sessionStorage` の runId。tabs権限は使わない)
- **「中止」でそれまでに集めた分のCSVをつくる**(全損させない)

- ⏳ ExtensionPay(課金)の組み込みはMasahiroのアカウント作成後(下記「Pro機能の動作確認」で先に動作は見られる)

## 無料 / Pro の境界

| 機能 | 無料 | Pro |
|---|---|---|
| 表示中ページの索引簿CSV(キャンセル除外・要確認判定) | ✅ | ✅ |
| 領収書ページからの金額・日付の自動補完 | ✅ | ✅ |
| **全ページ巡回**(年間分をまとめて) | — | ✅ |
| **領収書の一括保存**(電帳法のファイル名規約) | — | ✅ |

**領収書はHTMLで保存する**。Amazonは領収書をPDFで配信しておらず、拡張がPDFを生成することも
できないため。保存対象は領収書ページ(=電子取引データそのもの)で、電帳法の保存要件は満たす。
PDFが要る場合は保存したHTMLを開いて印刷→PDFに保存する。**掲載文でPDFを約束しないこと**。

## アーキテクチャ: リモートセレクタJSON

DOM構造の知識は全て `selectors.default.json` 側にあり、コードは抽出エンジンに徹する。
起動時に `https://keiri-tools.com/ext/amazon-receipt/selectors.json` を取得
(24hキャッシュ→古いキャッシュ→同梱デフォルトの順でフォールバック)。
**AmazonのDOM変更は docs/ext/amazon-receipt/selectors.json の更新+pushだけで直せる**
(ストア再審査不要。JSONは宣言的データでありコードではないため、MV3のリモートコード禁止に
抵触しない)。

抽出は候補リストの先勝ち: `css`(セレクタ+属性+regex) → `cardText`(カード全文に
ラベル文言regex)。クラス名は変わっても「注文日」「合計」の文言は変わりにくいので、
cardTextが最後の砦として機能する。

## 動作確認(Masahiro向け)

1. `chrome://extensions` → デベロッパーモードON → 「パッケージ化されていない拡張機能を
   読み込む」→ このフォルダを選択
2. https://www.amazon.co.jp/gp/css/order-history を開く → 右下にパネルが出る
3. 「このページをスキャン」→ 件数/合計が正しいか確認 → 「索引簿CSV」でDL
4. うまく取れないときは、ページを **ファイル→保存(完全)** で
   `~/Desktop/amazon_fixtures/` に保存してもらえれば、そのHTMLでセレクタを調整します

### Pro機能の動作確認(課金の組み込み前に試す)

ExtensionPayはまだ組み込んでいないので、ライセンスを手で立てて確認する。
注文履歴ページで **右クリック→検証→Console** を開き、以下を実行してページを再読み込み:

```js
chrome.storage.local.set({ license: { pro: true, source: "dev" } })
```

パネルの右上に「Pro」が出て、主ボタンが「一覧表をつくる（全ページ分）」になる。

- **「一覧表をつくる（全ページ分）」**: **実際にページが移動していく**(利用者がクリックするのと同じ)。
  1ページごとに「◯ページ目を読んでいます…」と出て、最後のページまで行ったらCSVを保存する。
  1.2秒間隔・安全上の上限30ページ。期間の絞り込み(例: 2025年)をしてから押すと速い。
  **途中で「中止」を押せば、それまでに集めた分でCSVができる**
- **「領収書もまとめて保存する」**: `ダウンロード/Amazon領収書/` に `日付_amazon_金額_注文番号.html` で保存
  (領収書ページの取得は fetch のままでよい — こちらは実績があり動いている)

元に戻す: `chrome.storage.local.remove("license")`

## content.js(ページ内スクリプト)をどう検査するか

単体テストが見ているのは `src/lib/*.js` の純ロジックだけで、**content.js は素通し**になる。
そこに「ボタンを押す → 状態を保存 → 移動 → 次のページで再開 → CSVを保存」という配線があり、
**機械の目が一度も通っていなかったから、壊れた巡回をそのまま出荷した**。

→ `tests/test_extension_crawl.mjs` が、**content.js の実コードを jsdom に読み込み、
`location.href` への代入を捕まえて次のページを読み込み直す**(＝ブラウザのページ移動の再現)。
`chrome.storage.local` / `sessionStorage` はページ移動をまたいで残るシムにしてあるので、
状態の持ち回りごと検査できる。**注文履歴を `fetch` したら即座に落ちる**番人も入れてある
(裏取得への逆戻りを防ぐ)。

### ⚠️ tools/store のハーネスは現在**壊れている**

`tools/store/harness.html` は 2026-07-13 のUI刷新より前のボタンID(`#kt-scan` `#kt-crawl`
`#kt-status`…)を押しており、いまの content.js では動かない(`node tools/store/shoot.mjs` は落ちる)。
直すときは、巡回シーンは**ページ移動を伴う**ので、静的な1ページのハーネスでは再現できないことに注意
(fetchシムでページを配る作りは、まさに本番で壊れた方式そのもの)。

```
bash tools/store/build_zip.sh   # ストアにアップロードするZIP(manifest参照の過不足も検査)
```

ストア掲載画像は**合成データのみ**。Masahiroの実注文履歴を使わない(個人データの公開になる)。

## ファイル名規約

`YYYYMMDD_amazon_金額_注文番号.html` — 索引簿の「保存ファイル名」列と一括保存の実ファイル名を
一致させる(検索性の担保)。索引簿CSVの列と保存ファイルが名前で突き合うので、税務調査時に
「この行の証憑はこのファイル」と示せる。

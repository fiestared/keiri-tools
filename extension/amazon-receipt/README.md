# アマゾン領収書・電帳法索引簿メーカー (Chrome拡張)

Amazon.co.jpの注文履歴から**電子帳簿保存法(電子取引)の索引簿CSV**を生成する拡張。
検索要件の3要素(取引年月日・取引金額・取引先)を必ず含み、ExcelでそのままJSON崩れなく
開ける(UTF-8 BOM+CRLF)。処理は全てブラウザ内で完結し、注文データは外部送信しない
(外部通信はセレクタ定義JSONの取得のみで、ユーザーデータは含まない)。

## 無料 / Pro の境界

| 機能 | 無料 | Pro |
|---|---|---|
| 表示中ページの索引簿CSV(キャンセル除外・要確認判定) | ✅ | ✅ |
| 領収書ページからの金額・日付の自動補完 | ✅ | ✅ |
| **全ページ巡回**(年間分をまとめて) | — | ✅ |
| **領収書の一括保存**(電帳法のファイル名規約) | — | ✅ |

**領収書はHTMLで保存する**。Amazonは領収書をPDFで配信しておらず、拡張がPDFを生成することも
できないため。保存対象は領収書ページ(=電子取引データそのもの)で、電帳法の保存要件は満たす。
PDFが要る場合は保存したHTMLを開いて印刷→PDFに保存する。**掲載文でPDFを約束しないこと**。

## アーキテクチャ: リモートセレクタJSON

DOM構造の知識は全て `selectors.default.json` 側にあり、コードは抽出エンジンに徹する。
起動時に `https://keiri-tools.com/ext/amazon-receipt/selectors.json` を取得
(24hキャッシュ→古いキャッシュ→同梱デフォルトの順でフォールバック)。
**AmazonのDOM変更は docs/ext/amazon-receipt/selectors.json の更新+pushだけで直せる**
(ストア再審査不要。JSONは宣言的データでありコードではないため、MV3のリモートコード禁止に
抵触しない)。

抽出は候補リストの先勝ち: `css`(セレクタ+属性+regex) → `cardText`(カード全文に
ラベル文言regex)。クラス名は変わっても「注文日」「合計」の文言は変わりにくいので、
cardTextが最後の砦として機能する。

## 動作確認(Masahiro向け)

1. `chrome://extensions` → デベロッパーモードON → 「パッケージ化されていない拡張機能を
   読み込む」→ このフォルダを選択
2. https://www.amazon.co.jp/gp/css/order-history を開く → 右下にパネルが出る
3. 「このページをスキャン」→ 件数/合計が正しいか確認 → 「索引簿CSV」でDL
4. うまく取れないときは、ページを **ファイル→保存(完全)** で
   `~/Desktop/amazon_fixtures/` に保存してもらえれば、そのHTMLでセレクタを調整します

### Pro機能の動作確認(課金の組み込み前に試す)

ExtensionPayはまだ組み込んでいないので、ライセンスを手で立てて確認する。
注文履歴ページで **右クリック→検証→Console** を開き、以下を実行してページを再読み込み:

```js
chrome.storage.local.set({ license: { pro: true, source: "dev" } })
```

パネルのタイトルが「v0.2 Pro」になり、🔒 が消える。

- **「全ページを巡回」**: 1.2秒間隔でページを送りながら全注文を集める(安全上の上限30ページ)。
  期間の絞り込み(例: 2025年)をしてから押すと速い。**同じ注文しか返らなくなった時点で自動的に止まる**
- **「領収書を一括保存」**: `ダウンロード/Amazon領収書/` に `日付_amazon_金額_注文番号.html` で保存

元に戻す: `chrome.storage.local.remove("license")`

## ファイル名規約

`YYYYMMDD_amazon_金額_注文番号.html` — 索引簿の「保存ファイル名」列と一括保存の実ファイル名を
一致させる(検索性の担保)。索引簿CSVの列と保存ファイルが名前で突き合うので、税務調査時に
「この行の証憑はこのファイル」と示せる。

## ストア公開時の注意(調査済みリスク)

- 名称の「アマゾン」: 商標リスクあり。ただし競合「アマゾン注文履歴フィルタ」(10万ユーザー)が
  同表記で長期運用中の前例あり。最終名は提出時に判断
- Chrome Web Store Payments は2022年廃止 → Pro課金はExtensionPay/Gumroad等の外部決済
- プライバシー実務: ユーザーデータの外部送信なしを単一目的説明に明記(審査で最重要)
- 競合との差別化 = **索引簿の自動生成**(既存はDL/印刷補助のみで索引簿未対応)
